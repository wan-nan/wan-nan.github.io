源码解读

  1. 变量声明初始化

    声明并初始化current_varno, new_varno, rt_length,sub_action,sub_action_ptr

  2. 制作rule action 和 qual 的可修改副本

    （这里传的是存储在 relcache 中的版本，所以千万不要对其进行修改操作）

  3. 获取必要的锁并修复任何已删除的 JOIN RTE 条目

    函数AcquireRewriteLocks()的解析请阅读笔者的另一篇文章：SQL引擎-查询重写-rewriteHandler.cpp解读（二）

  4. 调整规则 action 和 qual 以抵消其 varnos，以便我们可以将其 rtable 与主解析树的 rtable 合并

  5. 生成扩展 rtable，它由主解析树的 rtable 加上rule action的 rtable 组成；
    它将成为rule action的完整 rtable。

    在我们完成重写后，有些条目可能没有使用，但出于以下两个原因，会将它们全部保留在原位：

    • 如果我们有选择地删除 RT 条目，将很难调整查询的 varnos

    • 如果规则是 INSTEAD，则根本不会执行原始查询，因此必须保留其 rtable，以便执行程序对其进行正确的权限检查。

      未在完整连接树中引用的 RT 条目将被planner忽略，因此它们不会影响查询语义。但是其中指定的任何权限检查都将在执行程序启动期间应用（请参阅 ExecCheckRTEPerms()函数，该函数将在后续进行解读）。这允许我们在视图在结果查询中根本没有语义引用时，检查调用者是否具有视图的插入权限。

    • 当规则不是 INSTEAD 时，对其复制的 RT 条目进行的权限检查与执行原始查询期间进行的检查相比将是多余的，但这里没有必要去区分处理这种情况。

    注意：因为planner会彻底改变 rtable，我们必须确保rule action的 rtable 是独立的，并且不与主 rtable 共享子结构。因此在这里做一个深拷贝(deep copy)

  6. 解析树的 rtable 中可能有一些子链接，因此我们需要对 sub_action 做出相应的处理

  7. 另外，还需要将那些具有 RLS 条件的 RTE 归入到 sub_action 中

    并将其标记为hasRowSecurity，不论这些RTE在我们重写完成后是否会被引用

    （注意：目前这是一个no-op操作，因为 RLS 条件直到后面才会被添加，但看样子提前做总没错）

  8. 每个rule action的jointree应该由主解析树的jointree加上该规则的jointree构成，但通常不包含我们要替换的原始 rtindex（如果有rtindex存在，它也不会用于 INSERT）

    注意，如果rule action引用 OLD，则其jointree将添加对 rt_index 的引用

    如果rule action没有引用 OLD，但是 rule_qual 或用户查询 quals 引用了，那么我们需要在 jointree 中保留原始 rtindex 来为 quals 提供数据。

    但是，原始 rtindex 不能被join两次，所以如果rule action提到它会将其去除

    如上所述，**action的jointree不得与主解析树共享子结构**

  9. 如果原始查询有任何 CTE，需要它们复制到规则操作(rule action)中。

    但对于utility action来讲，则不需要上述操作

  10. Event Qualification 强制复制解析树并将其拆分为两个查询，一个 w/rule_qual，一个 w/NOT rule_qual

    并将用户查询qual添加到rule action

  11. 用合适的insert/update中的字段名重写target-list条目的右侧的new.attribute w/

  12. 如果 rule_action 有 RETURNING 子句

    • 如果触发查询(triggering query)没有 RETURNING 子句，则执行以下操作之一
      • 将其丢弃
      • 重写它以满足触发查询的 RETURNING 子句的要求

    如果多个规则都具有 RETURNING 子句，则报错

到此，rewriteRuleAction()函数的源码解读结束。
